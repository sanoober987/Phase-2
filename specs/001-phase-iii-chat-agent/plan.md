## Summary

Extend the existing FastAPI + Next.js todo application with a conversational AI
interface. Users interact with a chat panel that routes natural-language task commands
through the OpenAI Agents SDK to a stateless MCP tool server, persisting all
conversation turns in Neon PostgreSQL and returning plain-language confirmations.

---

## Technical Context

**Language/Version**: Python 3.12 (backend), TypeScript 5 / Next.js 14+ (frontend)

**Primary Dependencies**:
- Backend: FastAPI 0.115, SQLModel 0.0.21, asyncpg 0.30, PyJWT 2.10,
  `openai-agents` (new), `mcp` (new)
- Frontend: Next.js 14+, React 18, better-auth, Tailwind CSS, SWR

**Storage**: Neon Serverless PostgreSQL via asyncpg вЂ” existing engine + two new tables

**Testing**: pytest + pytest-asyncio (backend), jest + React Testing Library (frontend)

**Performance Goals**: Chat endpoint p95 < 5s; task CRUD p95 < 200ms

**Constraints**: Stateless backend (no in-memory session). JWT required on every request.

**Scale/Scope**: Single user-scoped app; hundreds of messages per user

## Implementation Phases

### Phase A вЂ” Database Layer

**Goal**: Add Conversation and Message tables to the existing Neon database.

Tasks:
1. Create `backend/app/models/conversation.py`
   - SQLModel table: UUID PK, user_id (indexed), created_at, updated_at
   - Use `DateTime(timezone=True)` on all timestamp fields (existing pattern)
2. Create `backend/app/models/message.py`
   - SQLModel table: UUID PK, conversation_id FK (indexed), role (str),
     content (Text column), tool_calls (JSON column, nullable), created_at
3. Modify `backend/app/database.py` в†’ `init_db()`
   - Import `Conversation` and `Message` so `create_all` creates tables
4. Create `backend/app/services/chat_service.py`
   - `get_or_create_conversation(db, user_id, conversation_id?) в†’ Conversation`
   - `load_history(db, conversation_id, limit=20) в†’ list[Message]`
   - `append_messages(db, conversation_id, user_msg, assistant_msg, tool_calls) в†’ tuple`

**Acceptance**: `init_db()` creates both tables; chat_service unit tests pass.

### Phase B вЂ” MCP Tool Server

**Goal**: Stateless FastMCP server exposing 5 task management tools.

Tasks:
1. Create `backend/app/mcp_tools/server.py` with FastMCP instance
2. Implement `add_task(user_id, title, description="")` в†’ delegates to task_service
3. Implement `list_tasks(user_id, completed=None)` в†’ delegates to task_service
4. Implement `complete_task(user_id, task_id)` в†’ sets completed=True via task_service
5. Implement `update_task(user_id, task_id, title=None, description=None)`
6. Implement `delete_task(user_id, task_id)` в†’ delegates to task_service
7. Each tool: validate `user_id` is non-empty; return `{status, data, error}` JSON
8. Add `__main__` entrypoint: `mcp.run()`

**Security**: Every tool validates `user_id` is present and non-empty before any DB call.

**Acceptance**: All 5 tools pass unit tests with mocked DB session; cross-user access
returns error response (not exception).

### Phase C вЂ” AI Agent Integration

**Goal**: OpenAI Agents SDK orchestration layer connecting chat to MCP tools.

Tasks:
1. Add to `requirements.txt`: `openai-agents>=0.0.6`, `mcp>=1.3.0`
2. Create `backend/app/services/agent_service.py`
   - `run_agent(user_id, messages) в†’ tuple[str, list[dict]]`
   - Constructs `MCPServerStdio` pointing at `mcp_tools/server.py`
   - Builds `Agent` with system prompt embedding `user_id`
   - Calls `await Runner.run(agent, input=messages)`
   - Returns `(final_output_string, tool_calls_log)`
3. System prompt template:
   ```
   You are a helpful task management assistant for user {user_id}.
   Use the available tools to create, list, complete, update, or delete tasks.
   Always confirm what action you took. If intent is unclear, ask a clarifying question.
   Never delete tasks without explicit instruction.
   ```

**Acceptance**: Unit test with mocked MCP verifies agent returns non-empty reply string
and list of tool call dicts.

## Security Design

### JWT Flow

```
User logs in в†’ Better Auth issues JWT (signed with BETTER_AUTH_SECRET)
Frontend stores JWT in httpOnly cookie
Frontend calls POST /api/{user_id}/chat with Authorization: Bearer <token>
FastAPI get_current_user dependency verifies signature + expiry
current_user["sub"] == user_id path param checked в†’ 403 if mismatch
All DB queries use current_user["sub"] as user_id filter
MCP tools receive user_id in tool arguments; validate non-empty before any DB call
```

### User Isolation Points

1. `get_current_user` вЂ” token signature and expiry verification
2. `user_id` path param vs token `sub` check вЂ” prevents calling another user's chat
3. `get_or_create_conversation` вЂ” validates conversation belongs to requesting user
4. `load_history` вЂ” filtered by conversation_id (owned by user)
5. All 5 MCP tools вЂ” validate `user_id` param; task_service scopes all queries by user_id

## Risks and Mitigations

| Risk | Impact | Likelihood | Mitigation |
|------|--------|------------|------------|
| MCP stdio subprocess latency в†’ p95 > 5s | Poor UX | Medium | Benchmark; switch to in-process MCPServerSse if needed |
| Agent hallucinates task_id for wrong user | Incorrect mutation | Low | task_service returns 404 if task not owned by user; tool returns error to agent |
| Neon connection exhaustion under concurrent chat | 503 / timeout | Low | Existing `pool_recycle=300` and `pool_pre_ping=True`; add explicit pool_size limits |
